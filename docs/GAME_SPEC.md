# 킹메이커스: 민심 경매 - 이식용 단일 문서

이 문서는 `킹메이커스: 민심 경매`를 새로운 기술 스택으로 옮길 때 필요한 게임 규칙, 데이터 구조, 구현 원칙, 현재 완성도, 우선순위를 한 파일로 정리한 이식용 브리프다.

> **이 문서가 룰의 단일 진실(Single Source of Truth)이다.**
> CLAUDE.md, skills/*/SKILL.md, 코드 주석이 이 문서와 충돌하면 이 문서가 우선한다.
> 1~30장은 원본 브리프이며, 부록 A는 이식 과정에서 채운 보완 결정사항이다.

현재 기준:

- 룰 기준: v0.3
- 점수 기준: scoring v1
- 엔진 스키마 기준: schemaVersion `0.2`
- 1차 플레이 기준: 4인

## 1. 게임 정체성

`킹메이커스: 민심 경매`는 정치 브로커 보드게임이다.

플레이어는 후보자가 아니라 정치 브로커다. 각 라운드마다 브로커들은 후보에게 자금을 배분해 출마 후보를 만들고, 후보의 공약을 선택하며, 유권자에게 영향력을 심고, 광고/스캔들/정책 압박/단일화를 통해 선거 결과와 정책 지형을 조작한다.

승리 목표는 단순한 후보 당선이 아니다. 후보 당선은 큰 공개 점수원이지만, 유권자 장악, 정책 변화, 비밀 의제 달성이 최종 승리를 좌우한다.

핵심 루프:

```text
후보 경매 -> 공약 선택 -> 유권자 조작 -> 캠페인 액션 -> 단일화 -> 투표 -> 정책 변화 -> 점수 정산
```

## 2. 핵심 디자인 원칙

1. 플레이어는 후보자가 아니라 정치 브로커다.
2. 후보 당선은 가장 큰 공개 점수원이지만 유일한 승리 방법이면 안 된다.
3. 유권자 장악과 정책 조작은 최종 비밀 의제로 이어지는 장기 전략이어야 한다.
4. 시스템이 처리할 수 있는 단계는 자동 처리한다.
5. 플레이어가 결정해야 하는 순간에만 게임이 멈춘다.
6. UI는 룰을 계산하지 않고 `GameState`를 표시하며 `GameAction`만 dispatch한다.
7. 모든 룰 계산은 순수 엔진/reducer에서 처리한다.
8. 랜덤은 seed 기반 또는 명시적 RNG 주입으로 처리한다.

## 6. 정책 트랙

정책 트랙은 5개다. 모든 트랙은 `-2..2` 범위이며 시작값은 0이다.

| 트랙 | - 방향 | + 방향 |
| --- | --- | --- |
| economy | 복지 | 시장 |
| labor | 노동권 | 기업 자율 |
| society | 시민 자유 | 질서 |
| industry | 환경 | 성장 |
| foreign | 개방 | 보호 |

정책 변화 원천:

1. 당선 후보의 공약 정책 이동
2. 당선 후보 카드의 당선 효과
3. 공개 단일화 조건의 정책 양보
4. 정책 압박 비교

정책 압박은 트랙별로 `- 방향 압박 총합`과 `+ 방향 압박 총합`을 비교한다. 한쪽이 더 크면 그 방향으로 1칸 움직인다. 차이가 큰 트랙부터 최대 2개 트랙만 한 라운드에 반영한다.

- 라운드 종료와 다음 라운드 진입

## 8. 후보 경매

기본 경매는 `동시 비공개 입찰`이다.

- 각 플레이어는 자기 입찰만 본다.
- 여러 후보에게 money를 나누어 입찰할 수 있다.
- 모든 플레이어가 입찰을 확정하면 시스템이 동시에 공개하고 정산한다.
- 총 입찰액 상위 3명이 출마한다.

공개 순차 입찰은 튜토리얼/디버그 모드로만 남긴다.

출마 후보 동점 처리:

1. 후원자 수가 많은 후보
2. 최고 입찰자 reputation이 높은 후보
3. 후보 id 기준 결정적 순서

돈 처리:

- 출마 후보에게 건 돈은 전액 소비한다.
- 탈락 후보에게 건 돈은 절반 환급한다.
- 환급은 내림 처리한다.

예: 탈락 후보에게 3 money를 걸었다면 1 money를 돌려받고 실제 비용은 2 money다.

캠프 슬롯:

- `majorBacker`: 해당 후보 최고 입찰자
- `coBacker`: majorBacker 제외, 2 money 이상 입찰한 최고 후원자
- `organizer`: 6-7인에서만 사용. 앞 슬롯 제외 2 money 이상 입찰한 최고 후원자

majorBacker는 해당 후보의 공약을 선택하고, 당선 시 가장 큰 점수를 얻는다.

## 9. 공약 선택

경매가 끝나면 바로 캠페인으로 가지 않고 공약 선택 단계에서 멈춘다.

- 각 출마 후보마다 공약 후보 3장이 제시된다.
- 해당 후보의 majorBacker가 1장을 선택한다.
- 공약은 정책 방향, 유권자 반응, 즉시 효과를 결정한다.
- 선택하지 않은 공약 후보는 버려진다.
- 모든 출마 후보 공약이 확정되면 유권자를 공개하고 캠페인 액션으로 넘어간다.

예외 처리:

- majorBacker가 없는 후보는 시스템이 자동으로 공약을 고른다.
- 공약 후보가 비정상적으로 비어 있으면 남은 덱에서 fallback으로 선택한다.

공약 효과 유형:

- 후보에게 추가 표를 준다.
- 특정 유권자 그룹이 공개되어 있으면 후보에게 추가 표를 준다.
- majorBacker에게 money 또는 VP를 준다.
- majorBacker에게 특정 유권자 그룹 영향력을 준다.
- majorBacker에게 정책 압박 토큰을 준다.

공약 효과는 한 후보/공약 조합당 한 번만 적용된다.

## 10. 유권자와 영향력

유권자 카드는 공개되면 영향력 마커를 받을 수 있다.

유권자 통제:

- 영향력이 가장 높은 플레이어가 단독 1등이면 그 유권자를 통제한다.
- 공동 1등이면 아무도 통제하지 못한다.
- 영향력 0이면 아무도 통제하지 못한다.

통제된 유권자는 `배치`로 특정 출마 후보에게 배정할 수 있다.

배치 규칙:

- 캠페인 액션 단계에서만 가능하다.
- 해당 유권자를 단독으로 통제하는 플레이어만 배치할 수 있다.
- 일반 캠페인 액션을 소모하지 않는 무료 보조 권한이다.
- 같은 유권자는 투표 전까지 다시 배치할 수 있다.
- 마지막 배치가 적용된다.
- 배치하지 않은 통제 유권자는 자동 투표 규칙을 따른다.

자동 투표:

- 통제자가 배치한 유권자는 배치된 후보에게 표를 준다.
- 배치되지 않은 유권자는 선호 태그와 맞는 공약을 가진 후보를 찾는다.
- 해당 후보가 없으면 기권한다.

표 변화:

| 관계 | 표 변화 |
| --- | ---: |
| 선호 태그와 맞음 | +1 |
| 중립 | 0 |
| 싫어하는 태그와 맞음 | -1 |
| 직접 충돌 태그 | -2 |

표는 0 아래로 내려가지 않는다.

라운드 종료 시 표를 행사한 통제 유권자는 플레이어에게 해당 유권자 그룹 영향력 마커를 남긴다. 이 마커는 최종 비밀 의제 점수에 사용한다.

## 11. 캠페인 액션

각 플레이어는 라운드마다 캠페인 액션 2개를 가진다. 진행 순서는 라운드로빈이다.

4인 예:

```text
Broker A -> Broker B -> Broker C -> Broker D -> Broker A -> Broker B -> Broker C -> Broker D
```

액션 목록:

| 액션 | 비용 | 액션 소모 | 효과 |
| --- | --- | --- | --- |
| 접촉 | organization 1 | 예 | 유권자 1명에게 내 영향력 +2 |
| 광고 | money 2 | 예 | 출마 후보 1명에게 +2표 |
| 압박 | organization 2 | 예 | 정책 트랙/방향에 내 압박 +1 |
| 모금 | reputation 1 | 예 | money +4 |
| 스캔들 | money 2, reputation 1 | 예 | 출마 후보 1명에게 -2표 |
| 조건지지 | 없음 | 예 | 출마 후보 1명에게 +1표. 당선 시 내게 +1 VP |
| 배치 | 없음 | 아니오 | 내가 통제하는 유권자를 출마 후보 1명에게 배정 |

현재 MVP 활성 후보 능력:

- 모금 money +1
- 노동자 유권자 접촉 비용 -1
- 정책 압박 비용 -1

공약 제한 1종, 공약 선택 시 평판 손실 약점 일부도 활성이다. 그 외 후보 능력/약점은 설명만 표시하고 실제 효과는 후순위로 둔다.

## 12. 단일화

단일화는 두 출마 후보 중 한 명이 사퇴하고 다른 한 명에게 표 일부를 넘기는 단계다.

흐름:

1. 캠페인 액션이 모두 끝나면 단일화 단계에 진입한다.
2. 출마 후보의 majorBacker가 단일화를 제안할 수 있다.
3. 제안자는 대표 후보와 사퇴 후보를 지정한다.
4. 사퇴 후보의 majorBacker가 수락하거나 거절한다.
5. 수락하면 사퇴 후보는 물러나고 계산된 표 일부가 대표 후보에게 이전된다.
6. 제안하지 않거나 거절되면 해당 제안자는 이번 라운드 단일화를 넘긴다.
7. 모든 가능한 제안자가 넘기거나 단일화가 성사되면 시스템이 투표로 진행한다.

이전 비율:

| 조건 | 이전 비율 |
| --- | ---: |
| 공약 정책 트랙과 방향이 같음 | 70% |
| 지지 유권자 그룹이 일부 겹침 | 50% |
| 후보 성향이 충돌하지 않음 | 30% |
| 공약/성향이 직접 충돌함 | 불가 |

이전되는 표는 사퇴 후보의 현재 표에 이전 비율을 곱한 뒤 내림한다.

현재 구현 범위:

- 공개 단일화 제안
- 수락
- 거절
- 스킵

후속 규칙:

- 공개 조건 계약
- 비밀 pact
- 계약 이행/배신 처리
- 공개 계약 이행 점수

비밀 pact 배신 규칙 일부:

- 배신자 reputation -2
- 피해자 유권자 이벤트 2장 획득
- 배신자에게 `betrayed-secret-pact` 플래그 기록

하지만 비밀 협상 UI와 실제 플레이 흐름은 아직 없다.

## 13. 투표

투표는 시스템이 해결한다.

후보의 최종 표는 다음을 합산한다.

1. 후보 기본 표
2. 공약 효과 표
3. 광고, 스캔들, 조건지지 등 캠페인 표
4. 유권자 배정 표
5. 단일화 이전 표

가장 많은 표를 얻은 후보가 당선된다.

투표 동점 처리:

1. majorBacker의 reputation
2. 후보의 총 후원금
3. 표를 준 유권자 그룹 수
4. 후보 id 기준 결정적 순서

## 14. 정책 변화

투표 후 정책 변화가 해결된다.

정책 변화 순서:

1. 당선 후보의 공약 정책 이동
2. 당선 후보 카드의 당선 효과
3. 공개 단일화 조건에 정책 양보가 있으면 적용
4. 정책 압박 비교

후보 당선 효과 유형:

- `fixedPolicyMove`: 시스템이 자동 적용
- `choosePolicyMove`: majorBacker가 옵션 중 선택
- `flexPolicyMove`: majorBacker가 트랙/방향 선택
- `winningPromisePolicyMove`: 당선 공약 정책 이동 반복 적용

선택형/자유형 효과는 `electionEffectSelection` 단계에서 처리한다. majorBacker가 없거나 선택 옵션이 비정상적이면 엔진 fallback으로 처리한다.

## 15. 라운드 점수

현재 엔진 구현값은 scoring v1 기준이다.

| 조건 | 점수 |
| --- | ---: |
| 당선 후보 majorBacker | +4 VP |
| 당선 후보 coBacker | +2 VP |
| 당선 후보 organizer | +1 VP |
| 당선 후보 공약 실행 보너스 majorBacker | +1 VP |
| 2위 후보 majorBacker | +1 VP |
| 조건지지 성공 | +1 VP |
| 내 정책 압박이 실제 반영됨 | +1 VP |
| 내가 통제한 유권자가 당선 후보를 지지함 | +1 VP |
| 6-7인 킹메이커 점수 | +2 VP |

통제 유권자 지지 VP 상한:

- 4-5인: 플레이어당 라운드 최대 1 VP
- 6-7인: 플레이어당 라운드 최대 2 VP

6-7인 킹메이커 점수:

- 후보 간 표 차이가 2 이하일 때
- majorBacker가 아닌 플레이어가 당선 후보에게 유권자 표를 많이 제공하면 +2 VP

라운드 점수 정산 후, 표를 행사한 통제 유권자의 그룹별 영향력 마커가 플레이어에게 누적된다. 이 마커는 최종 비밀 의제 점수에 사용한다.

## 16. 최종 점수와 승리

5라운드 종료 후 최종 점수가 가장 높은 플레이어가 승리한다.

최종 점수 구성:

1. 라운드 중 공개 VP
2. 비밀 의제 점수
3. 남은 money 5마다 +1 VP
4. reputation 단독 1등 +2 VP
5. 공개 계약 이행 수 단독 1등 +2 VP

동점 처리 v0.3 기준:

1. victoryPoints가 높은 플레이어
2. reputation이 높은 플레이어
3. 남은 money가 많은 플레이어
4. 영향력 마커 총합이 많은 플레이어
5. 그래도 같으면 공동 승리

목표 점수대:

- 4인 5라운드 최종 승자: 30-45 VP

## 17. 비밀 의제

비밀 의제는 게임의 장기 목표다. 플레이어별 비밀 정보로 다뤄야 하며, 5라운드 전체 기록을 참조할 수 있어야 한다.

조건 유형:

- `policyAtLeast`: 특정 정책 트랙이 특정 방향/값 이상
- `policyBetween`: 특정 정책 트랙이 범위 안에 있음
- `noExtremePolicies`: 극단 정책 없음
- `influenceLeader`: 특정 유권자 그룹 영향력 1위
- `influenceLeaderAny`: 여러 그룹 중 하나 영향력 1위
- `notInfluenceLeader`: 특정 그룹 영향력 1위가 아님
- `candidateWon`: 특정 후보 당선
- `candidateWonAtMost`: 특정 후보 당선 횟수 제한
- `reputationAtLeast`: reputation 일정 이상
- `reputationRankAtMost`: reputation 순위
- `remainingMoneyAtLeast`: 남은 money 일정 이상

비밀 의제는 현재 라운드 상태만 보면 안 된다. 전체 라운드 히스토리, 누적 영향력 마커, 정책 최종 상태, 후보 당선 이력을 참조해야 한다.

현재 비밀 의제별 이론상 최대 점수는 대략 8-13 VP 범위다.

## 21. 방/멀티플레이 흐름

KLUON식 방 흐름 기준:

1. 호스트가 Kingmakers 방 생성
2. 초대 링크 공유
3. 참가자는 이름 입력 후 로비 입장
4. 참가자는 `준비 완료` 클릭
5. 모두 준비되면 호스트가 `게임 시작`
6. 시작 시 시스템이 좌석을 랜덤 배정
7. 모든 참가자는 자기 좌석 개인 화면으로 게임 진행
8. 공용 테이블 화면은 진행 상황과 공개 정보를 보여줌

로비는 참가자 목록, 준비 상태, 시작 버튼만 보여준다. 인게임 보드와 액션은 별도 플레이 화면에서 보여준다.

URL/화면 개념:

- table/host 화면: 공개 진행자 화면, 시스템 진행 가능
- player 화면: 자기 비밀 의제와 자기 액션만 가능
- spectator 화면: 공개 정보 관전

## 23. 현재 완성도

현재 상태는 "룰 엔진 검증용 MVP"로는 꽤 진행되어 있다.

- 4인 로컬 플레이 흐름: 높음
- 후보 경매와 공약 선택: 중상
- 유권자 영향력과 캠페인 액션: 중상
- 투표/정책/라운드 점수 계산: 중상
- 5라운드 풀게임 밸런스: 낮음
- 비밀 정보/협상/이벤트 경험: 낮음
- 상용 보드게임 룰북 완성도: 초안 단계

정량 감각:

- 디지털 룰 엔진 프로토타입: 약 70%
- 사람에게 설명 가능한 완성 보드게임: 약 45-55%

## 24. 강점

1. 핵심 루프가 명확하다.
2. 플레이어가 후보자가 아니라 브로커라는 정체성이 규칙에 반영되어 있다.
3. 후보 당선과 정책 지형이 분리되어 전략 목표가 여러 층으로 생긴다.
4. 경매의 majorBacker/coBacker 구조가 플레이어 간 이해관계를 만든다.
5. 유권자 통제와 정책 압박이 단순 표 싸움 이상의 선택지를 만든다.
6. 엔진 테스트와 타입 구조가 있어 규칙 변경을 검증하기 좋다.

## 25. 문제점과 TODO

1. 초보자 온보딩이 부족하다.
2. 동시 비공개 경매가 실제로 재미있는지 공개 순차 모드와 비교 테스트해야 한다.
3. 단일화 계약/비밀 pact가 아직 완성되지 않았다.
4. 이벤트 카드 효과가 대부분 TODO다.
5. 비밀 의제가 실제 UX에서 충분히 비밀스럽지 않다.
6. 후보 능력/약점 대부분이 아직 실제 룰로 작동하지 않는다.
7. `배치`가 무료 보조 권한이라 유권자 통제 점수 상한을 계속 관리해야 한다.
8. 점수 소스가 많아 초반에 읽기 어렵다.
9. 1라운드 재미와 5라운드 재미가 다를 수 있다.
10. 로그와 카드 설명이 아직 개발자 친화적이다.
11. 승리 목표와 라운드 종료 조건이 화면에서 충분히 강조되어야 한다.

## 27. 새 스택 개발 프롬프트

아래 프롬프트를 새 프로젝트의 초기 구현 지시문으로 사용할 수 있다.

```text
킹메이커스: 민심 경매를 새로운 스택으로 구현한다.

이 게임은 정치 브로커 보드게임이다. 플레이어는 후보자가 아니라 브로커이며, 후보를 후원하고 공약을 선택하고 유권자와 정책을 조작해 승점을 얻는다.

우선 UI보다 순수 룰 엔진을 먼저 만든다. GameState, GameAction, reducer, 카드 데이터, phase 자동 진행을 분리한다. UI는 GameState를 표시하고 GameAction을 dispatch만 해야 하며, 룰 계산을 UI에 넣지 않는다.

첫 목표는 4인 1라운드가 끝까지 플레이되는 것이다. 이후 5라운드 풀게임, 비밀 의제 최종 점수, 동점 판정, 플레이어별 비밀 화면을 추가한다.

반드시 구현해야 할 핵심 루프는 다음이다:

수입 -> 이슈 공개 -> 후보 공개 -> 동시 비공개 후보 경매 -> 출마 후보 3명 확정 -> majorBacker 공약 선택 -> 유권자 공개 -> 캠페인 액션 2회씩 -> 단일화 -> 투표 -> 정책 변화 -> 라운드 점수 -> 다음 라운드/최종 점수.

기본 데이터 수량은 후보 21장, 공약 30장, 유권자 48장, 이슈 15장, 후보 이벤트 30장, 유권자 이벤트 30장, 비밀 의제 16장이다.

이벤트 효과 대부분은 후순위로 두고, 첫 구현은 경매/공약/유권자/정책/점수에 집중한다.

시스템 액션과 플레이어 액션을 분리한다. 시스템이 자동 처리할 수 있는 단계는 자동 진행하고, 플레이어 결정이 필요한 순간에만 멈춘다.

방 흐름은 로비와 인게임을 분리한다. 로비는 참가자 목록, ready, host start만 보여준다. host가 시작하면 좌석은 랜덤 배정되고 각 플레이어는 자기 개인 화면에서만 자기 액션과 비밀 의제를 본다.
```

## 28. 최소 테스트 목록

새 스택에서 반드시 테스트해야 할 항목:

- seed가 같으면 setup 결과가 같다.
- 4/5/6/7인 설정이 정확하다.
- 후보 공개 수가 인원별 설정과 맞다.
- 동시 비공개 입찰 후 상위 3명이 출마한다.
- 경매 동점 처리가 결정적이다.
- 출마 후보 입찰금은 소비되고 탈락 후보 입찰금은 절반 환급된다.
- majorBacker/coBacker/organizer가 올바르게 배정된다.
- majorBacker가 공약을 선택한다.
- 공약 fallback이 작동한다.
- 유권자 영향력 단독 1등만 controller가 된다.
- 공동 1등이면 controller가 없다.
- 배치는 액션을 소모하지 않는다.
- 캠페인 액션은 플레이어당 2회다.
- 광고/스캔들/조건지지 표 변화가 반영된다.
- 정책 압박은 최대 2개 트랙만 움직인다.
- 단일화 이전 비율이 30/50/70%로 계산된다.
- 투표 동점 처리가 결정적이다.
- 라운드 점수 이유가 플레이어별로 남는다.
- 유권자 통제 VP 상한이 인원별로 적용된다.
- 최종 비밀 의제가 라운드 히스토리를 참조한다.
- 최종 money/reputation/public contract 보너스가 적용된다.
- UI는 룰 계산 없이 GameState와 action log 기반으로 표시한다.

## 29. MVP에서 일부러 미룰 것

첫 이식 MVP에서는 아래를 과감히 미룬다.

- 모든 이벤트 카드 효과 구현
- 모든 후보 능력/약점 구현
- 정교한 협상 UI
- 비밀 pact 전체 UX
- 상용 수준 애니메이션
- 복잡한 AI/bot
- 전체 밸런스 확정

먼저 4인 1라운드가 명확하고 재미있게 끝나는 것이 중요하다.

## 30. 가장 중요한 결론

이 게임을 완성하려면 기능을 더 많이 붙이는 것보다 다음 세 가지가 먼저 명확해야 한다.

1. 플레이어는 이번 단계에서 무엇을 해야 하는가.
2. 그 선택은 어떤 점수 루트로 이어지는가.
3. 5라운드가 끝났을 때 무엇이 승패를 갈랐는가.

이 세 가지가 룰 엔진, 화면, 룰북에서 동시에 설명되면 `킹메이커스: 민심 경매`는 디버그 프로토타입에서 실제 플레이 가능한 전략 보드게임으로 넘어갈 수 있다.

## 부록 A. 이식 보완 결정사항 (원본 브리프에 없어서 채운 것)

이 부록은 원본 브리프가 정하지 않은 부분에 대한 이식 결정이다. 원본 1~30장과 충돌하지 않는 범위에서만 유효하다.

### A-1. 기술 스택 확정
- **TypeScript 단일 언어 모노레포** (pnpm workspaces). 브리프의 데이터 모델(§18)과 reducer/dispatch 원칙(§2)이 TS 기준이므로 엔진을 서버·클라이언트가 그대로 공유한다.
  - `packages/engine` — 순수 룰 엔진 (외부 의존성 0)
  - `packages/data` — 카드 데이터 + zod 스키마
  - `apps/server` — Node 20+, Express + Socket.IO (권위 서버)
  - `apps/client` — React 18 + Vite + Zustand
- 테스트: Vitest. §28 최소 테스트 목록은 전부 엔진 단위 테스트로 작성한다.

### A-2. 계정 없음 — KLUON식 초대 링크
- 회원가입/DB 인증 없음. 방 생성 시 `roomId` 발급, 초대 링크(`/room/:roomId`)로 입장, 이름만 입력한다.
- 입장 시 서버가 `playerToken`(uuid)을 발급하고 클라이언트 localStorage에 저장 — 새로고침/재접속 시 이 토큰으로 좌석을 복원한다.
- MVP 저장소는 서버 메모리(Map). 방은 휘발성이며, 액션 로그가 있으므로 추후 영속화(파일/DB)는 로그 저장만 붙이면 된다.

### A-3. 뷰 마스킹 규약
- 서버만 reducer를 실행한다(권위 서버). 클라이언트에는 `projectView(state, viewer)`를 거친 마스킹 상태만 전송한다.
- viewer 종류: `table`(공개 정보 + 시스템 진행), `player-{seat}`(공개 정보 + 자기 비밀), `spectator`(공개 정보만).
- 마스킹 대상: 타인 비밀 의제, `auctionBidding` 중 타인 입찰 내용(확정 여부만 공개), 타인 이벤트 카드 손패. **CSS 숨김이 아니라 데이터 자체를 보내지 않는다.**

### A-4. 미정 액션의 잠정 처리
- `poll`(여론조사): 브리프에 비용/효과 미정. 액션 타입과 constants 자리만 만들고 **비활성 플래그**로 둔다. (잠정안: money 2, 액션 소모, 아직 비공개 정보가 없어 효과 보류)
- `useEvent`: 이벤트 카드는 손패 수집 + 설명 표시까지만. 사용 시 "효과 미구현" 안내 후 거부. 7단계에서 활성화.
- `generateRandomBids`: 디버그/튜토리얼 전용 시스템 액션으로 유지.

### A-5. 결정성 규약
- 엔진 내부에서 `Date.now()`, `Math.random()` 금지. RNG는 seed 기반 함수(mulberry32)를 GameState에 상태로 보관해 주입한다.
- 모든 동점 처리(§8, §13, §16)는 마지막 단계가 id/좌석 기준 결정적 순서로 끝나야 한다.
- 같은 seed + 같은 액션 시퀀스 재생 = 같은 최종 상태 (리플레이/재접속 복원의 근거).

### A-6. 액션 로그 규약
- 모든 reducer 처리 결과는 `ActionLogEntry`로 남긴다: `{ seq, phase, actor, action, summary, effects[] }`.
- `effects[]`는 "누구에게 무엇이 변했는지"의 구조화 목록 — §22의 "로그를 읽지 않아도 방금 일어난 일이 이해되어야 한다"를 UI가 이걸로 구현한다.
- `summary`는 한국어 자연어 문장으로 엔진이 생성한다 (예: "브로커 B가 청년 유권자에게 영향력 +2").

### A-7. 비밀 의제 배분 방식
- §17은 의제가 "플레이어별 비밀 정보"라고만 규정하고 배분 절차(장수 선택 여부)는 명시하지 않는다.
- 잠정 결정: setup 시 의제 덱에서 플레이어당 **1장을 무작위로** 배분한다 (2장 중 1장 선택하는 절차 없음). MVP 단순성 우선이며, 추후 "2장 중 선택" 방식으로 바꾸더라도 setup.ts 내부 배분 로직만 교체하면 되고 GameState 계약(`secretAgendaId: AgendaId | null`)은 유지된다.

### A-8. 유권자 그룹 명칭
- 브리프가 명시적으로 언급하는 유권자 그룹은 §11의 "노동자"뿐이다. 나머지 그룹 구성은 브리프 미명시 사항이다.
- 결정: 48장을 6그룹 × 8장으로 균등 배분한다 — 노동자, 중산층, 자영업자, 청년층, 노년층, 농어촌 (`packages/data/src/tags.ts`의 `VoterGroupKey`).
- 각 유권자 카드는 그룹과 별개로 카드 단위 선호/비선호/직접충돌 태그(§10)를 가지므로, 그룹 소속 자체가 투표 계산에 직접 개입하지 않는다 — 그룹은 §15/§17의 "그룹별 영향력 마커 누적·최종 의제 판정" 집계 단위로만 쓰인다.

### A-9. 공약 반응 태그 = 정책 트랙 파생 10종
- §9의 "유권자 반응"과 §10의 "선호/비선호/직접 충돌 태그"가 참조하는 태그 어휘를 브리프가 별도로 정의하지 않는다.
- 결정: 정책 트랙 5종 × 방향 2종에서 기계적으로 파생되는 10개 태그로 통일한다 (`economy-welfare`/`economy-market` 등, `packages/data/src/tags.ts`의 `PolicyReactionTag`). 공약의 `reactionTag`는 자신의 `policyMove.track/direction`에서 그대로 계산되고, 유권자의 `conflictTag`는 `preferredTag`의 정반대 태그(같은 트랙, 반대 방향)로 계산된다. 태그 어휘를 이원화하지 않음으로써 "공약 정책 트랙과 방향이 같음"(§12)과 "선호 태그와 맞음"(§10) 판정이 동일한 비교 로직을 공유할 수 있다.

### A-10. 유권자 카드의 표 수(voteWeight)
- §13은 "유권자 배정 표"를 최종 표 합산의 한 요소로 규정하지만, 유권자 카드 1장이 몇 표에 해당하는지는 브리프에 수치가 없다.
- 결정: 각 유권자 카드에 `voteWeight`(1~3, `packages/data/src/voters.ts`)를 부여한다. 최종 표 기여량은 A-11의 관계 계수를 곱해 계산한다.

### A-11. §10 "표 변화" 표의 적용 대상 재해석
- §10 원문의 표(선호 태그와 맞음 +1 / 중립 0 / 싫어하는 태그와 맞음 -1 / 직접 충돌 태그 -2)가 정확히 **어느 유권자에게** 적용되는지 브리프가 특정하지 않는다.
- 미배치 유권자는 §10에서 "선호 태그와 맞는 공약을 가진 후보를 찾는다. 없으면 기권한다"고만 규정하므로, 관계는 항상 "선호"(+1)이거나 아예 표를 주지 않는(기권) 둘 중 하나다 — 표의 중립/싫어함/충돌 행이 적용될 여지가 없다.
- 결정: 표 전체(중립/싫어함/충돌 포함)는 **배치된(assignVoterChoice) 유권자**에게 적용된다. 컨트롤러는 통제한 유권자를 자기 선호와 무관한 후보에게도 배정할 수 있지만, 그 배정이 유권자 본인의 선호/비선호/직접충돌 태그와 실제로 어떤 관계인지에 따라 표가 늘거나(+1) 그대로거나(0) 깎이거나(-1) 아예 역효과가 난다(-2, 0 미만은 클램프). "유권자를 장악해도 아무 후보에게나 자유롭게 표를 몰 수 없다"는 원 기획 의도(도입부 pitch)와 부합해 채택한 해석이다.
- 최종 표 기여 공식: `배치된 유권자 = voteWeight × relation(0 미만 클램프)`, `미배치 유권자(매칭 성공 시) = voteWeight × 1`. `relation`은 배치 후보가 선택한 공약의 `reactionTag`와 유권자의 `preferredTag`/`dislikedTag`/`conflictTag`를 비교해 정한다.

### A-12. 단일화 이전 비율 판정 — "성향 충돌"을 공약 정책 방향 충돌로 통일
- §12 이전 비율 표의 4단계 중 "공약 정책 트랙과 방향이 같음"(70%)과 "공약/성향이 직접 충돌함"(불가)을 **같은 축**(대표 후보·사퇴 후보가 각각 선택한 공약의 policyMove)으로 판정한다: 두 공약이 같은 트랙+같은 방향이면 70%, 같은 트랙+반대 방향이면 직접 충돌(불가), 트랙 자체가 다르면 다음 단계로 내려간다.
- "지지 유권자 그룹이 일부 겹침"(50%)은 두 후보 카드의 `supportedVoterGroups` 교집합 존재 여부로 판정한다.
- "후보 성향이 충돌하지 않음"(30%)은 위 두 조건 중 어느 쪽도 해당하지 않을 때의 기본값이다 — 직접 충돌 여부는 이미 공약 트랙 비교 단계에서 걸러졌으므로, 이 지점에 도달했다는 것 자체가 "충돌하지 않음"을 의미한다.
- `CandidateCard.leaningTags`는 이 계산에 사용하지 않는다 — 브리프가 태그 간 대립 관계를 정의하지 않아, 별도 매핑을 새로 만드는 대신 이미 존재하는 공약 트랙 비교로 대체했다. leaningTags는 카드 플레이버로 유지하며 후속 스킬이 필요하면 활용할 수 있다.
- **단일화는 라운드당 최대 1건**: "모든 가능한 제안자가 넘기거나 단일화가 성사되면 투표로 진행한다"를 "성사 즉시 해당 라운드의 단일화 단계를 종료한다"로 해석했다. 성사 후 연쇄로 추가 단일화를 허용하는지는 브리프에 명시되지 않아, 더 단순한 "최대 1건"을 채택했다.

### A-13. 서버가 모든 액션 뒤에 runUntilPlayerAction까지 자동으로 이어 진행한다
- §19는 "runUntilPlayerAction은 자동 처리 가능한 단계를 넘기고 플레이어 결정 지점에서 멈춘다"만 정의하고, 누가/언제 그 호출을 트리거하는지는 명시하지 않았다. §22는 "플레이어가 액션해야 할 때만 화면이 멈춰야 한다"를 필수 UX로 못박는다.
- 이 두 요구를 그대로 두면 모순이 생긴다: `placeBid`의 마지막 확정처럼 개별 플레이어 액션 하나가 자동 진행 phase(예: `auctionResolved`)에 도착시킬 수 있는데, 시스템 액션은 호스트만 보낼 수 있으므로(§20) 호스트가 "진행" 버튼을 누르기 전까지 그 방의 모든 참가자 화면이 실제로는 아무도 결정할 것이 없는데도 멈춰 있게 된다 — §22 위반.
- 결정: `apps/server/src/sockets.ts`의 `game:action` 핸들러가 매 액션 처리 직후 서버 스스로 `{ type: 'runUntilPlayerAction' }`을 이어서 `reduce`하고, 그 결과 상태·로그를 함께 브로드캐스트한다. 클라이언트(어떤 역할이든)가 별도로 진행을 트리거할 필요가 없다.
- `applyRunUntilPlayerAction`은 이미 결정 지점/게임 종료 상태에서도 안전하게 no-op으로 성공 처리하도록 설계되어 있어 항상 이어붙여도 부작용이 없다.
- TableScreen의 "진행" 버튼(skills/client-lobby-table)은 §19가 명시한 대로 UI 요소로는 유지하되(플레이어 결정 대기 중엔 비활성), 정상 동작 중에는 서버가 이미 최신 상태를 보내주므로 사실상 클릭할 필요가 없는 안전망/수동 재동기화 용도가 된다.

### A-14. 후보/공약 덱 소진 방지 — 당선/선택되지 않은 카드는 덱으로 돌아간다
- 브리프는 5개 덱(후보/공약/유권자/이슈/이벤트) 중 어느 것도 소진 시 처리(재셔플/버림 더미 등)를 명시하지 않는다. 직접 계산해보면 **후보와 공약 덱만 실제로 문제가 된다**: 4인 기준 5라운드 동안 라운드당 공개 후보 5장 × 5라운드 = 25장 필요(보유 21장), 공약은 출마 후보 3명 × 3장 제시 × 5라운드 = 45장 필요(보유 30장) — 둘 다 완주 전에 바닥난다. 유권자(48장 중 25장 소모)·이슈(15장 중 5장)는 문제없고, 이벤트 덱은 이미 `deck.shift()`가 `undefined`를 안전하게 처리하므로(부록 A-4로 어차피 미사용) 손댈 필요가 없다.
- 결정: **라운드가 끝나고 cleanup에서 다음 라운드로 넘어갈 때(`applyNextRound`), 이번 라운드에 공개됐지만 "소비되지 않은" 카드를 각자의 덱 맨 아래로 되돌린다.**
  - 후보: 이번 라운드에 공개된 후보(`candidatesRevealed`) 중 **당선자(`winnerCandidateId`)를 제외한 전원**이 덱으로 돌아간다 — 경매에서 탈락한 후보, 출마했지만 낙선한 후보, 단일화로 사퇴한 후보를 모두 포함한다. 라운드당 실제로 영구 소모되는 후보는 당선자 1명뿐이므로 인원수·라운드 수와 무관하게 21장으로 항상 충분하다.
  - 공약: 각 출마 후보에게 제시된 공약 3장(`promiseOptions`) 중 **선택되지 않은 나머지**가 덱으로 돌아간다. 라운드당 영구 소모는 출마 후보 수(항상 3명)만큼만 발생하므로 5라운드 기준 15장으로 30장 안에서 항상 충분하다.
  - 되돌리는 순서는 라운드 종료 시점 상태에서 결정적으로 계산되는 배열 순서를 그대로 쓴다(추가 셔플 없음, RNG 상태를 소비하지 않음) — 반환된 카드는 다음번에 다시 뽑힐 때 그 순서대로(맨 아래 → 나중에 다시 위로) 나온다. 완전 무작위 재셔플이 아니라 "덱 맨 아래로" 방식을 택한 이유는 결정성 재생(부록 A-5)을 해치지 않기 위함이다.
  - 후보가 낙선/사퇴 후 다시 공개되어 재출마하고 재당선하는 것은 규칙 위반이 아니다 — 오히려 §17 `candidateWonAtMost` 조건("특정 후보가 N회를 넘게 당선되면 안 된다")이 의미를 가지려면 같은 후보가 여러 라운드에 걸쳐 반복 출마할 수 있어야 하므로, 이 결정은 그 조건과도 일관적이다.

### A-15. 단일화 "공개 조건 계약" — 대표 후보 당선을 조건으로 자동 집행
> **주의: Phase 7(고급 규칙)은 4인 실제 플레이테스트 없이 사용자가 명시적으로 게이트를 건너뛰고 착수했다.** 이 항목과 A-16은 특히 실제 플레이 검증이 없는 상태로 구현되었으므로, 추후 플레이테스트에서 규칙 자체가 조정될 가능성이 A-1~A-14보다 높다.

- §12는 "공개 조건 계약"을 후속 규칙으로만 언급하고(§12, §25-3), 계약이 정확히 무엇을 담는지는 skills/advanced-rules/SKILL.md의 "자금 이전, VP 약속, 정책 양보(policy.ts 3순서 훅 연결)"라는 항목 나열 외에는 정의하지 않는다. §16-5는 "공개 계약 이행 수 단독 1등 +2 VP"만 명시한다.
- 결정: `proposeUnification`에 선택적 `terms: UnificationTerm[]` (moneyTransfer / victoryPointGrant / policyConcession 3종, SKILL.md가 나열한 항목과 정확히 대응)을 추가했다. 수락되면 `UnificationContract`로 기록되고, **대표 후보가 실제로 당선됐을 때만** `policyResolution`(§14 4순서 중 브리프가 이미 "단일화 양보" 자리로 비워둔 ③번 훅)에서 자동으로 이행된다. 낙선하면 계약은 조용히 무효가 된다 — "불이행"이라는 별도 상태/페널티 경로는 만들지 않았다(§16-5 "이행 수"는 성사된 것만 세면 되므로, 실패를 벌하는 규칙을 새로 만들 필요가 없다고 판단).
- VP 약속(`victoryPointGrant`)은 제안자의 VP를 깎지 않는 순수 지급으로 처리했다 — VP는 이 게임에서 소비 가능한 자원으로 쓰인 적이 없어(§5, §15 어디에도 VP를 대가로 치르는 규칙이 없음), "이전"이 아니라 "보상 지급"으로 해석하는 편이 기존 자원 모델과 일관적이다.
- 자금 이전(`moneyTransfer`)만 제안 시점에 제안자의 보유 자금으로 상한을 검증한다 — VP·정책 양보는 자금처럼 "바닥날 수 있는" 자원이 아니므로 같은 검증이 적용되지 않는다.
- `contractsFulfilled`는 라운드로 리셋되지 않는 게임 전체 누계 필드로 PlayerState에 추가했다 — §16-5가 "게임 종료 시점의 이행 수"를 묻고 있어 라운드 단위 필드로는 부족했다.

### A-16. 비밀 pact — "지지 약속" 하나로 범위를 좁히고, 이행 여부는 실제 행동으로 자동 판정
> A-15와 같은 주의 사항이 적용된다 — 플레이테스트 없이 구현됨.

- §12는 비밀 pact에 대해 "당사자만 아는 pact 체결"과 배신 시 페널티 3종(배신자 reputation -2, 피해자 유권자 이벤트 2장, `betrayed-secret-pact` 플래그)만 명시하고, **"하지만 비밀 협상 UI와 실제 플레이 흐름은 아직 없다"**고 브리프 스스로 밝힌다. 약속의 내용, 제안 가능 시점, "배신"을 판정하는 방법 전부가 미정이다.
- 결정: 약속의 내용을 **"지정한 출마 후보를 지지하겠다"(`supportCandidate`) 한 가지**로 좁혔다. 이유: 라운드 종료 시점에 "실제로 지켰는가"를 별도 UI(예: 명시적 "배신하기" 버튼) 없이도 이미 존재하는 행동 기록만으로 판정할 수 있는 유일한 약속 형태이기 때문이다 — 자금·정책 등 다른 약속 유형은 "지켰다"의 기준이 불명확해 별도 확인 절차가 필요해지고, 그러면 브리프가 아직 없다고 명시한 "협상 UI"를 새로 설계해야 한다.
- 제안·수락·거절은 `campaignActions` phase 중 언제든(차례 무관, 비용 없음) 가능하게 했다 — §11의 `assignVoterChoice`("무료 보조 권한")와 성격이 같다고 봤다.
- 이행 판정(`roundScoring` 시점, §15 라운드 점수 정산과 같은 타이밍): 제안자가 약속한 후보의 majorBacker/coBacker 본인이거나, 그 후보에게 `conditionalSupport`를 실행한 기록이 있으면 이행으로 본다. 그 외(다른 후보를 지지했거나 캠페인에 전혀 관여하지 않음)는 전부 배신으로 판정한다 — "부분 이행"이나 "정상 참작" 같은 중간 상태는 두지 않았다.
- 배신 페널티 3종은 §12 명세를 그대로 따른다: 제안자 reputation -2, 상대방이 유권자 이벤트 2장 획득(`voterEventHand`를 PlayerState에 신설 — 기존 `candidateEventHand`와 별도 관리), 제안자에게 `betrayedSecretPact` 불리언 플래그(한 번이라도 배신하면 게임이 끝날 때까지 true로 고정, §12가 "플래그"라고 표현해 카운터가 아닌 1회성 마킹으로 해석).
- 마스킹: `pendingSecretPactProposals`/`activeSecretPacts`는 당사자(제안자·상대방) 외 모든 시점(타 플레이어·table·spectator)에서 배열 자체가 비워진다. 관련 액션 로그 3종(`proposeSecretPact`/`acceptSecretPact`/`declineSecretPact`)도 summary에 상대·후보를 담지 않을 뿐 아니라, 당사자가 아닌 뷰에서는 로그 항목 자체를 제거한다 — actor 필드만으로도 "누가 비밀 행동을 했는지"가 새어나가는 것을 막기 위함이다(CLAUDE.md: 비밀 정보는 데이터 자체를 보내지 않는다).
- 배신 후 페널티(reputation 변화, 상대의 카드 획득)는 공개 정보이므로 마스킹하지 않는다 — pact의 존재/내용만 비밀이고, 그 결과로 발생하는 자원 변화는 다른 자원 변화와 동일하게 공개된다.

### A-17. 이벤트 카드 효과 60장 — 소수의 재사용 가능한 효과 유형 + 손패 상한 없음 + campaignActions 중 무료 사용
> A-15와 같은 주의 사항이 적용된다 — 플레이테스트 없이 구현됨.

- §29(MVP 유예 목록)·부록 A-4는 이벤트 카드 효과 60장 전체를 `{ todo: true }`로 미루면서 "손패 상한·사용 타이밍 규칙 확정 필요"라고만 남겼다. 카드 60장의 헤드라인(예: "노동절 집회", "후원금 논란")은 순수 플레이버 텍스트이고 기계적 수치를 전혀 내포하지 않는다.
- 결정: 카드마다 고유 효과를 만드는 대신, `PromiseEffect`(5종/30장)·`CandidateAbility`(6종/21장)와 같은 기존 관례를 따라 **재사용 가능한 효과 3유형**을 정의하고 카드마다 파라미터만 다르게 배정했다.
  - `resourceDelta`: 사용자 자신의 money/organization/reputation을 증감(양수·음수 모두 가능) — 헤드라인에 특정 후보가 지목되지 않는 후보 이벤트(30장) 대부분에 사용. "이 뉴스를 내가 어떻게 활용/방어하는가"로 해석했다.
  - `candidateVotesDelta`: 사용 시 지정하는 출마 후보(신설된 `useEvent.targetCandidateId`)의 `campaignVotes`를 증감 — 후보 이벤트 중 "지지 선언", "지지층 결집" 등 선거 판세에 직접 개입하는 헤드라인에 배정.
  - `groupInfluence`: 지정된 유권자 그룹에 속한 **공개된 유권자 전원**에게 사용자의 영향력을 +N — 유권자 이벤트(30장)는 전부 헤드라인이 특정 그룹(노동자/청년층/노년층/농어촌/자영업자/중산층)을 명시하고 있어 30장 전부 이 유형으로 배정했다. 전부 "그 그룹이 부각되는 시사 이슈"라 효과는 항상 양수로 통일했다(부정적 유권자 이벤트라는 개념 자체를 만들지 않음).
- **손패 상한: 없음.** 라운드당 최대 1장씩 들어오고(§5 수입), 5라운드 기준으로도 손패가 5장을 넘지 않아 관리 부담이 크지 않다고 판단해 별도 상한 규칙을 추가하지 않았다.
- **사용 타이밍: `campaignActions` phase 중 언제든, 차례·비용 무관.** `assignVoterChoice`·비밀 pact 제안과 같은 "무료 보조 권한" 성격으로 통일했다 — 이벤트 카드를 캠페인 액션 2회 중 하나로 소모하게 하면 다른 7종 액션보다 항상 비효율적이 되어 사실상 죽은 자원이 될 위험이 있었다.
- `useEvent`는 `candidateEventHand`와 `voterEventHand`(부록 A-16에서 신설) 양쪽에서 eventId를 찾아 소모한다 — 카드가 어느 손패에서 왔는지에 따라 조회할 카탈로그(`candidateEvents`/`voterEvents`)만 달라질 뿐 처리 로직은 동일하다.

### A-18. 후보 능력 2종 실제 집행 + poll(여론조사) 활성화
> A-15와 같은 주의 사항이 적용된다 — 플레이테스트 없이 구현됨.

- **후보 능력 집행 공백 발견**: `CandidateAbility`의 6유형 중 `fundraiseBonus`/`voterContactDiscount`/`policyPressureDiscount`는 이미 `campaign.ts`에 실제로 연결되어 있었지만, `promiseRestriction`(candidate-09 "서다영")과 `reputationLossOnPromise`(candidate-05 "한지수")는 타입·카드 데이터·zod 스키마까지 전부 존재하면서도 `promise.ts`의 `applySelectPromise`가 후보의 `abilities`를 아예 조회하지 않아 **실제로는 아무 효과도 없었다** — 카드에 적혀 있는 약점이 게임에 반영되지 않는 조용한 버그였다.
  - 결정: `applySelectPromise`에서 **선택 대상 후보 자신의** `abilities`만 확인한다(campaign.ts의 액션 능력들처럼 "내가 미는 모든 후보"를 모으지 않는다) — `promiseRestriction`/`reputationLossOnPromise`는 그 후보 개인의 특성이지, 같이 후원 중인 다른 후보에게 전이될 이유가 없기 때문이다.
  - `promiseRestriction`: 제시된 공약의 `reactionTag`가 `excludedTag`와 같으면 선택 자체를 거부한다(액션 검증 실패, `{ok:false}`).
  - `reputationLossOnPromise`: 공약이 확정되는 순간 majorBacker의 reputation을 `amount`만큼 즉시 차감하고 effects[]에 사유를 남긴다.
- **poll(여론조사) 활성화**: 부록 A-4가 남긴 잠정안(money 2, 액션 소모)을 그대로 채택했다. 효과는 브리프에 없어 새로 정했다 — **현재 출마 후보들의 예상 득표(`computeVoteBreakdown` 총합)를 로그 summary로 전원에게 공개**한다. 실제 여론조사가 결과를 공표하는 것과 같은 맥락이라 요청자만 보는 비공개 정보로 만들지 않았다. `campaignActions`의 나머지 6개 유료 액션과 동일하게 `runTurnAction`(차례·비용 검증, 액션 1회 소모)을 그대로 재사용한다.
