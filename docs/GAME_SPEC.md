# 킹메이커스: 민심 경매 - 이식용 단일 문서

이 문서는 `킹메이커스: 민심 경매`를 새로운 기술 스택으로 옮길 때 필요한 게임 규칙, 데이터 구조, 구현 원칙, 현재 완성도, 우선순위를 한 파일로 정리한 이식용 브리프다.

> **이 문서가 룰의 단일 진실(Single Source of Truth)이다.**
> CLAUDE.md, skills/*/SKILL.md, 코드 주석이 이 문서와 충돌하면 이 문서가 우선한다.
> 1~30장은 원본 브리프이며, 부록 A는 이식 과정에서 채운 보완 결정사항이다.

현재 기준:

- 룰 기준: v0.3
- 점수 기준: scoring v1
- 엔진 스키마 기준: schemaVersion `0.2`
- 1차 플레이 기준: 4인
- 풀게임 기준: 5라운드

## 1. 게임 정체성

`킹메이커스: 민심 경매`는 정치 브로커 보드게임이다.

플레이어는 후보자가 아니라 정치 브로커다. 각 라운드마다 브로커들은 후보에게 자금을 배분해 출마 후보를 만들고, 후보의 공약을 선택하며, 유권자에게 영향력을 심고, 광고/스캔들/정책 압박/단일화를 통해 선거 결과와 정책 지형을 조작한다.

승리 목표는 단순한 후보 당선이 아니다. 후보 당선은 큰 공개 점수원이지만, 유권자 장악, 정책 변화, 비밀 의제 달성이 최종 승리를 좌우한다.

핵심 루프:

```text
후보 경매 -> 공약 선택 -> 유권자 조작 -> 캠페인 액션 -> 단일화 -> 투표 -> 정책 변화 -> 점수 정산
```

## 2. 핵심 디자인 원칙

1. 플레이어는 후보자가 아니라 정치 브로커다.
2. 후보 당선은 가장 큰 공개 점수원이지만 유일한 승리 방법이면 안 된다.
3. 유권자 장악과 정책 조작은 최종 비밀 의제로 이어지는 장기 전략이어야 한다.
4. 시스템이 처리할 수 있는 단계는 자동 처리한다.
5. 플레이어가 결정해야 하는 순간에만 게임이 멈춘다.
6. UI는 룰을 계산하지 않고 `GameState`를 표시하며 `GameAction`만 dispatch한다.
7. 모든 룰 계산은 순수 엔진/reducer에서 처리한다.
8. 랜덤은 seed 기반 또는 명시적 RNG 주입으로 처리한다.

## 6. 정책 트랙

정책 트랙은 5개다. 모든 트랙은 `-2..2` 범위이며 시작값은 0이다.

| 트랙 | - 방향 | + 방향 |
| --- | --- | --- |
| economy | 복지 | 시장 |
| labor | 노동권 | 기업 자율 |
| society | 시민 자유 | 질서 |
| industry | 환경 | 성장 |
| foreign | 개방 | 보호 |

정책 변화 원천:

1. 당선 후보의 공약 정책 이동
2. 당선 후보 카드의 당선 효과
3. 공개 단일화 조건의 정책 양보
4. 정책 압박 비교

정책 압박은 트랙별로 `- 방향 압박 총합`과 `+ 방향 압박 총합`을 비교한다. 한쪽이 더 크면 그 방향으로 1칸 움직인다. 차이가 큰 트랙부터 최대 2개 트랙만 한 라운드에 반영한다.

- 모든 캠페인 액션 완료 후 투표/정책/점수 정산
- 라운드 종료와 다음 라운드 진입

## 8. 후보 경매

기본 경매는 `동시 비공개 입찰`이다.

- 각 플레이어는 자기 입찰만 본다.
- 여러 후보에게 money를 나누어 입찰할 수 있다.
- 모든 플레이어가 입찰을 확정하면 시스템이 동시에 공개하고 정산한다.
- 총 입찰액 상위 3명이 출마한다.

공개 순차 입찰은 튜토리얼/디버그 모드로만 남긴다.

출마 후보 동점 처리:

1. 후원자 수가 많은 후보
2. 최고 입찰자 reputation이 높은 후보
3. 후보 id 기준 결정적 순서

돈 처리:

- 출마 후보에게 건 돈은 전액 소비한다.
- 탈락 후보에게 건 돈은 절반 환급한다.
- 환급은 내림 처리한다.

예: 탈락 후보에게 3 money를 걸었다면 1 money를 돌려받고 실제 비용은 2 money다.

캠프 슬롯:

- `majorBacker`: 해당 후보 최고 입찰자
- `coBacker`: majorBacker 제외, 2 money 이상 입찰한 최고 후원자
- `organizer`: 6-7인에서만 사용. 앞 슬롯 제외 2 money 이상 입찰한 최고 후원자

majorBacker는 해당 후보의 공약을 선택하고, 당선 시 가장 큰 점수를 얻는다.

## 9. 공약 선택

경매가 끝나면 바로 캠페인으로 가지 않고 공약 선택 단계에서 멈춘다.

- 각 출마 후보마다 공약 후보 3장이 제시된다.
- 해당 후보의 majorBacker가 1장을 선택한다.
- 공약은 정책 방향, 유권자 반응, 즉시 효과를 결정한다.
- 선택하지 않은 공약 후보는 버려진다.
- 모든 출마 후보 공약이 확정되면 유권자를 공개하고 캠페인 액션으로 넘어간다.

예외 처리:

- majorBacker가 없는 후보는 시스템이 자동으로 공약을 고른다.
- 공약 후보가 비정상적으로 비어 있으면 남은 덱에서 fallback으로 선택한다.

공약 효과 유형:

- 후보에게 추가 표를 준다.
- 특정 유권자 그룹이 공개되어 있으면 후보에게 추가 표를 준다.
- majorBacker에게 money 또는 VP를 준다.
- majorBacker에게 특정 유권자 그룹 영향력을 준다.
- majorBacker에게 정책 압박 토큰을 준다.

공약 효과는 한 후보/공약 조합당 한 번만 적용된다.

## 10. 유권자와 영향력

유권자 카드는 공개되면 영향력 마커를 받을 수 있다.

유권자 통제:

- 영향력이 가장 높은 플레이어가 단독 1등이면 그 유권자를 통제한다.
- 공동 1등이면 아무도 통제하지 못한다.
- 영향력 0이면 아무도 통제하지 못한다.

통제된 유권자는 `배치`로 특정 출마 후보에게 배정할 수 있다.

배치 규칙:

- 캠페인 액션 단계에서만 가능하다.
- 해당 유권자를 단독으로 통제하는 플레이어만 배치할 수 있다.
- 일반 캠페인 액션을 소모하지 않는 무료 보조 권한이다.
- 같은 유권자는 투표 전까지 다시 배치할 수 있다.
- 마지막 배치가 적용된다.
- 배치하지 않은 통제 유권자는 자동 투표 규칙을 따른다.

자동 투표:

- 통제자가 배치한 유권자는 배치된 후보에게 표를 준다.
- 배치되지 않은 유권자는 선호 태그와 맞는 공약을 가진 후보를 찾는다.
- 해당 후보가 없으면 기권한다.

표 변화:

| 관계 | 표 변화 |
| --- | ---: |
| 선호 태그와 맞음 | +1 |
| 중립 | 0 |
| 싫어하는 태그와 맞음 | -1 |
| 직접 충돌 태그 | -2 |

표는 0 아래로 내려가지 않는다.

라운드 종료 시 표를 행사한 통제 유권자는 플레이어에게 해당 유권자 그룹 영향력 마커를 남긴다. 이 마커는 최종 비밀 의제 점수에 사용한다.

## 11. 캠페인 액션

각 플레이어는 라운드마다 캠페인 액션 2개를 가진다. 진행 순서는 라운드로빈이다.

4인 예:

```text
Broker A -> Broker B -> Broker C -> Broker D -> Broker A -> Broker B -> Broker C -> Broker D
```

액션 목록:

| 액션 | 비용 | 액션 소모 | 효과 |
| --- | --- | --- | --- |
| 접촉 | organization 1 | 예 | 유권자 1명에게 내 영향력 +2 |
| 광고 | money 2 | 예 | 출마 후보 1명에게 +2표 |
| 압박 | organization 2 | 예 | 정책 트랙/방향에 내 압박 +1 |
| 모금 | reputation 1 | 예 | money +4 |
| 스캔들 | money 2, reputation 1 | 예 | 출마 후보 1명에게 -2표 |
| 조건지지 | 없음 | 예 | 출마 후보 1명에게 +1표. 당선 시 내게 +1 VP |
| 배치 | 없음 | 아니오 | 내가 통제하는 유권자를 출마 후보 1명에게 배정 |

현재 MVP 활성 후보 능력:

- 모금 money +1
- 노동자 유권자 접촉 비용 -1
- 정책 압박 비용 -1

공약 제한 1종, 공약 선택 시 평판 손실 약점 일부도 활성이다. 그 외 후보 능력/약점은 설명만 표시하고 실제 효과는 후순위로 둔다.

## 12. 단일화

단일화는 두 출마 후보 중 한 명이 사퇴하고 다른 한 명에게 표 일부를 넘기는 단계다.

흐름:

1. 캠페인 액션이 모두 끝나면 단일화 단계에 진입한다.
2. 출마 후보의 majorBacker가 단일화를 제안할 수 있다.
3. 제안자는 대표 후보와 사퇴 후보를 지정한다.
4. 사퇴 후보의 majorBacker가 수락하거나 거절한다.
5. 수락하면 사퇴 후보는 물러나고 계산된 표 일부가 대표 후보에게 이전된다.
6. 제안하지 않거나 거절되면 해당 제안자는 이번 라운드 단일화를 넘긴다.
7. 모든 가능한 제안자가 넘기거나 단일화가 성사되면 시스템이 투표로 진행한다.

이전 비율:

| 조건 | 이전 비율 |
| --- | ---: |
| 공약 정책 트랙과 방향이 같음 | 70% |
| 지지 유권자 그룹이 일부 겹침 | 50% |
| 후보 성향이 충돌하지 않음 | 30% |
| 공약/성향이 직접 충돌함 | 불가 |

이전되는 표는 사퇴 후보의 현재 표에 이전 비율을 곱한 뒤 내림한다.

현재 구현 범위:

- 공개 단일화 제안
- 수락
- 거절
- 스킵

후속 규칙:

- 공개 조건 계약
- 비밀 pact
- 계약 이행/배신 처리
- 공개 계약 이행 점수

비밀 pact 배신 규칙 일부:

- 배신자 reputation -2
- 피해자 유권자 이벤트 2장 획득
- 배신자에게 `betrayed-secret-pact` 플래그 기록

하지만 비밀 협상 UI와 실제 플레이 흐름은 아직 없다.

## 13. 투표

투표는 시스템이 해결한다.

후보의 최종 표는 다음을 합산한다.

1. 후보 기본 표
2. 공약 효과 표
3. 광고, 스캔들, 조건지지 등 캠페인 표
4. 유권자 배정 표
5. 단일화 이전 표

가장 많은 표를 얻은 후보가 당선된다.

투표 동점 처리:

1. majorBacker의 reputation
2. 후보의 총 후원금
3. 표를 준 유권자 그룹 수
4. 후보 id 기준 결정적 순서

## 14. 정책 변화

투표 후 정책 변화가 해결된다.

정책 변화 순서:

1. 당선 후보의 공약 정책 이동
2. 당선 후보 카드의 당선 효과
3. 공개 단일화 조건에 정책 양보가 있으면 적용
4. 정책 압박 비교

후보 당선 효과 유형:

- `fixedPolicyMove`: 시스템이 자동 적용
- `choosePolicyMove`: majorBacker가 옵션 중 선택
- `flexPolicyMove`: majorBacker가 트랙/방향 선택
- `winningPromisePolicyMove`: 당선 공약 정책 이동 반복 적용

선택형/자유형 효과는 `electionEffectSelection` 단계에서 처리한다. majorBacker가 없거나 선택 옵션이 비정상적이면 엔진 fallback으로 처리한다.

## 15. 라운드 점수

현재 엔진 구현값은 scoring v1 기준이다.

| 조건 | 점수 |
| --- | ---: |
| 당선 후보 majorBacker | +4 VP |
| 당선 후보 coBacker | +2 VP |
| 당선 후보 organizer | +1 VP |
| 당선 후보 공약 실행 보너스 majorBacker | +1 VP |
| 2위 후보 majorBacker | +1 VP |
| 조건지지 성공 | +1 VP |
| 내 정책 압박이 실제 반영됨 | +1 VP |
| 내가 통제한 유권자가 당선 후보를 지지함 | +1 VP |
| 6-7인 킹메이커 점수 | +2 VP |

통제 유권자 지지 VP 상한:

- 4-5인: 플레이어당 라운드 최대 1 VP
- 6-7인: 플레이어당 라운드 최대 2 VP

6-7인 킹메이커 점수:

- 후보 간 표 차이가 2 이하일 때
- majorBacker가 아닌 플레이어가 당선 후보에게 유권자 표를 많이 제공하면 +2 VP

라운드 점수 정산 후, 표를 행사한 통제 유권자의 그룹별 영향력 마커가 플레이어에게 누적된다. 이 마커는 최종 비밀 의제 점수에 사용한다.

## 16. 최종 점수와 승리

5라운드 종료 후 최종 점수가 가장 높은 플레이어가 승리한다.

최종 점수 구성:

1. 라운드 중 공개 VP
2. 비밀 의제 점수
3. 남은 money 5마다 +1 VP
4. reputation 단독 1등 +2 VP
5. 공개 계약 이행 수 단독 1등 +2 VP

동점 처리 v0.3 기준:

1. victoryPoints가 높은 플레이어
2. reputation이 높은 플레이어
3. 남은 money가 많은 플레이어
4. 영향력 마커 총합이 많은 플레이어
5. 그래도 같으면 공동 승리

목표 점수대:

- 4인 5라운드 최종 승자: 30-45 VP

## 17. 비밀 의제

비밀 의제는 게임의 장기 목표다. 플레이어별 비밀 정보로 다뤄야 하며, 5라운드 전체 기록을 참조할 수 있어야 한다.

조건 유형:

- `policyAtLeast`: 특정 정책 트랙이 특정 방향/값 이상
- `policyBetween`: 특정 정책 트랙이 범위 안에 있음
- `noExtremePolicies`: 극단 정책 없음
- `influenceLeader`: 특정 유권자 그룹 영향력 1위
- `influenceLeaderAny`: 여러 그룹 중 하나 영향력 1위
- `notInfluenceLeader`: 특정 그룹 영향력 1위가 아님
- `candidateWon`: 특정 후보 당선
- `candidateWonAtMost`: 특정 후보 당선 횟수 제한
- `reputationAtLeast`: reputation 일정 이상
- `reputationRankAtMost`: reputation 순위
- `remainingMoneyAtLeast`: 남은 money 일정 이상

비밀 의제는 현재 라운드 상태만 보면 안 된다. 전체 라운드 히스토리, 누적 영향력 마커, 정책 최종 상태, 후보 당선 이력을 참조해야 한다.

현재 비밀 의제별 이론상 최대 점수는 대략 8-13 VP 범위다.

## 21. 방/멀티플레이 흐름

KLUON식 방 흐름 기준:

1. 호스트가 Kingmakers 방 생성
2. 초대 링크 공유
3. 참가자는 이름 입력 후 로비 입장
4. 참가자는 `준비 완료` 클릭
5. 모두 준비되면 호스트가 `게임 시작`
6. 시작 시 시스템이 좌석을 랜덤 배정
7. 모든 참가자는 자기 좌석 개인 화면으로 게임 진행
8. 공용 테이블 화면은 진행 상황과 공개 정보를 보여줌

로비는 참가자 목록, 준비 상태, 시작 버튼만 보여준다. 인게임 보드와 액션은 별도 플레이 화면에서 보여준다.

URL/화면 개념:

- table/host 화면: 공개 진행자 화면, 시스템 진행 가능
- player 화면: 자기 비밀 의제와 자기 액션만 가능
- spectator 화면: 공개 정보 관전

## 23. 현재 완성도

현재 상태는 "룰 엔진 검증용 MVP"로는 꽤 진행되어 있다.

- 4인 로컬 플레이 흐름: 높음
- 후보 경매와 공약 선택: 중상
- 유권자 영향력과 캠페인 액션: 중상
- 투표/정책/라운드 점수 계산: 중상
- 5라운드 풀게임 밸런스: 낮음
- 비밀 정보/협상/이벤트 경험: 낮음
- 상용 보드게임 룰북 완성도: 초안 단계

정량 감각:

- 디지털 룰 엔진 프로토타입: 약 70%
- 사람에게 설명 가능한 완성 보드게임: 약 45-55%

## 24. 강점

1. 핵심 루프가 명확하다.
2. 플레이어가 후보자가 아니라 브로커라는 정체성이 규칙에 반영되어 있다.
3. 후보 당선과 정책 지형이 분리되어 전략 목표가 여러 층으로 생긴다.
4. 경매의 majorBacker/coBacker 구조가 플레이어 간 이해관계를 만든다.
5. 유권자 통제와 정책 압박이 단순 표 싸움 이상의 선택지를 만든다.
6. 엔진 테스트와 타입 구조가 있어 규칙 변경을 검증하기 좋다.

## 25. 문제점과 TODO

1. 초보자 온보딩이 부족하다.
2. 동시 비공개 경매가 실제로 재미있는지 공개 순차 모드와 비교 테스트해야 한다.
3. 단일화 계약/비밀 pact가 아직 완성되지 않았다.
4. 이벤트 카드 효과가 대부분 TODO다.
5. 비밀 의제가 실제 UX에서 충분히 비밀스럽지 않다.
6. 후보 능력/약점 대부분이 아직 실제 룰로 작동하지 않는다.
7. `배치`가 무료 보조 권한이라 유권자 통제 점수 상한을 계속 관리해야 한다.
8. 점수 소스가 많아 초반에 읽기 어렵다.
9. 1라운드 재미와 5라운드 재미가 다를 수 있다.
10. 로그와 카드 설명이 아직 개발자 친화적이다.
11. 승리 목표와 라운드 종료 조건이 화면에서 충분히 강조되어야 한다.

## 27. 새 스택 개발 프롬프트

아래 프롬프트를 새 프로젝트의 초기 구현 지시문으로 사용할 수 있다.

```text
킹메이커스: 민심 경매를 새로운 스택으로 구현한다.

이 게임은 정치 브로커 보드게임이다. 플레이어는 후보자가 아니라 브로커이며, 후보를 후원하고 공약을 선택하고 유권자와 정책을 조작해 승점을 얻는다.

우선 UI보다 순수 룰 엔진을 먼저 만든다. GameState, GameAction, reducer, 카드 데이터, phase 자동 진행을 분리한다. UI는 GameState를 표시하고 GameAction을 dispatch만 해야 하며, 룰 계산을 UI에 넣지 않는다.

첫 목표는 4인 1라운드가 끝까지 플레이되는 것이다. 이후 5라운드 풀게임, 비밀 의제 최종 점수, 동점 판정, 플레이어별 비밀 화면을 추가한다.

반드시 구현해야 할 핵심 루프는 다음이다:

수입 -> 이슈 공개 -> 후보 공개 -> 동시 비공개 후보 경매 -> 출마 후보 3명 확정 -> majorBacker 공약 선택 -> 유권자 공개 -> 캠페인 액션 2회씩 -> 단일화 -> 투표 -> 정책 변화 -> 라운드 점수 -> 다음 라운드/최종 점수.

기본 데이터 수량은 후보 21장, 공약 30장, 유권자 48장, 이슈 15장, 후보 이벤트 30장, 유권자 이벤트 30장, 비밀 의제 16장이다.

이벤트 효과 대부분은 후순위로 두고, 첫 구현은 경매/공약/유권자/정책/점수에 집중한다.

시스템 액션과 플레이어 액션을 분리한다. 시스템이 자동 처리할 수 있는 단계는 자동 진행하고, 플레이어 결정이 필요한 순간에만 멈춘다.

방 흐름은 로비와 인게임을 분리한다. 로비는 참가자 목록, ready, host start만 보여준다. host가 시작하면 좌석은 랜덤 배정되고 각 플레이어는 자기 개인 화면에서만 자기 액션과 비밀 의제를 본다.
```

## 28. 최소 테스트 목록

새 스택에서 반드시 테스트해야 할 항목:

- seed가 같으면 setup 결과가 같다.
- 4/5/6/7인 설정이 정확하다.
- 후보 공개 수가 인원별 설정과 맞다.
- 동시 비공개 입찰 후 상위 3명이 출마한다.
- 경매 동점 처리가 결정적이다.
- 출마 후보 입찰금은 소비되고 탈락 후보 입찰금은 절반 환급된다.
- majorBacker/coBacker/organizer가 올바르게 배정된다.
- majorBacker가 공약을 선택한다.
- 공약 fallback이 작동한다.
- 유권자 영향력 단독 1등만 controller가 된다.
- 공동 1등이면 controller가 없다.
- 배치는 액션을 소모하지 않는다.
- 캠페인 액션은 플레이어당 2회다.
- 광고/스캔들/조건지지 표 변화가 반영된다.
- 정책 압박은 최대 2개 트랙만 움직인다.
- 단일화 이전 비율이 30/50/70%로 계산된다.
- 투표 동점 처리가 결정적이다.
- 라운드 점수 이유가 플레이어별로 남는다.
- 유권자 통제 VP 상한이 인원별로 적용된다.
- 최종 비밀 의제가 라운드 히스토리를 참조한다.
- 최종 money/reputation/public contract 보너스가 적용된다.
- UI는 룰 계산 없이 GameState와 action log 기반으로 표시한다.

## 29. MVP에서 일부러 미룰 것

첫 이식 MVP에서는 아래를 과감히 미룬다.

- 모든 이벤트 카드 효과 구현
- 모든 후보 능력/약점 구현
- 정교한 협상 UI
- 비밀 pact 전체 UX
- 상용 수준 애니메이션
- 복잡한 AI/bot
- 전체 밸런스 확정

먼저 4인 1라운드가 명확하고 재미있게 끝나는 것이 중요하다.

## 30. 가장 중요한 결론

이 게임을 완성하려면 기능을 더 많이 붙이는 것보다 다음 세 가지가 먼저 명확해야 한다.

1. 플레이어는 이번 단계에서 무엇을 해야 하는가.
2. 그 선택은 어떤 점수 루트로 이어지는가.
3. 5라운드가 끝났을 때 무엇이 승패를 갈랐는가.

이 세 가지가 룰 엔진, 화면, 룰북에서 동시에 설명되면 `킹메이커스: 민심 경매`는 디버그 프로토타입에서 실제 플레이 가능한 전략 보드게임으로 넘어갈 수 있다.

## 부록 A. 이식 보완 결정사항 (원본 브리프에 없어서 채운 것)

이 부록은 원본 브리프가 정하지 않은 부분에 대한 이식 결정이다. 원본 1~30장과 충돌하지 않는 범위에서만 유효하다.

### A-1. 기술 스택 확정
- **TypeScript 단일 언어 모노레포** (pnpm workspaces). 브리프의 데이터 모델(§18)과 reducer/dispatch 원칙(§2)이 TS 기준이므로 엔진을 서버·클라이언트가 그대로 공유한다.
  - `packages/engine` — 순수 룰 엔진 (외부 의존성 0)
  - `packages/data` — 카드 데이터 + zod 스키마
  - `apps/server` — Node 20+, Express + Socket.IO (권위 서버)
  - `apps/client` — React 18 + Vite + Zustand
- 테스트: Vitest. §28 최소 테스트 목록은 전부 엔진 단위 테스트로 작성한다.

### A-2. 계정 없음 — KLUON식 초대 링크
- 회원가입/DB 인증 없음. 방 생성 시 `roomId` 발급, 초대 링크(`/room/:roomId`)로 입장, 이름만 입력한다.
- 입장 시 서버가 `playerToken`(uuid)을 발급하고 클라이언트 localStorage에 저장 — 새로고침/재접속 시 이 토큰으로 좌석을 복원한다.
- MVP 저장소는 서버 메모리(Map). 방은 휘발성이며, 액션 로그가 있으므로 추후 영속화(파일/DB)는 로그 저장만 붙이면 된다.

### A-3. 뷰 마스킹 규약
- 서버만 reducer를 실행한다(권위 서버). 클라이언트에는 `projectView(state, viewer)`를 거친 마스킹 상태만 전송한다.
- viewer 종류: `table`(공개 정보 + 시스템 진행), `player-{seat}`(공개 정보 + 자기 비밀), `spectator`(공개 정보만).
- 마스킹 대상: 타인 비밀 의제, `auctionBidding` 중 타인 입찰 내용(확정 여부만 공개), 타인 이벤트 카드 손패. **CSS 숨김이 아니라 데이터 자체를 보내지 않는다.**

### A-4. 미정 액션의 잠정 처리
- `poll`(여론조사): 브리프에 비용/효과 미정. 액션 타입과 constants 자리만 만들고 **비활성 플래그**로 둔다. (잠정안: money 2, 액션 소모, 아직 비공개 정보가 없어 효과 보류)
- `useEvent`: 이벤트 카드는 손패 수집 + 설명 표시까지만. 사용 시 "효과 미구현" 안내 후 거부. 7단계에서 활성화.
- `generateRandomBids`: 디버그/튜토리얼 전용 시스템 액션으로 유지.

### A-5. 결정성 규약
- 엔진 내부에서 `Date.now()`, `Math.random()` 금지. RNG는 seed 기반 함수(mulberry32)를 GameState에 상태로 보관해 주입한다.
- 모든 동점 처리(§8, §13, §16)는 마지막 단계가 id/좌석 기준 결정적 순서로 끝나야 한다.
- 같은 seed + 같은 액션 시퀀스 재생 = 같은 최종 상태 (리플레이/재접속 복원의 근거).

### A-6. 액션 로그 규약
- 모든 reducer 처리 결과는 `ActionLogEntry`로 남긴다: `{ seq, phase, actor, action, summary, effects[] }`.
- `effects[]`는 "누구에게 무엇이 변했는지"의 구조화 목록 — §22의 "로그를 읽지 않아도 방금 일어난 일이 이해되어야 한다"를 UI가 이걸로 구현한다.
- `summary`는 한국어 자연어 문장으로 엔진이 생성한다 (예: "브로커 B가 청년 유권자에게 영향력 +2").

### A-7. 비밀 의제 배분 방식
- §17은 의제가 "플레이어별 비밀 정보"라고만 규정하고 배분 절차(장수 선택 여부)는 명시하지 않는다.
- 잠정 결정: setup 시 의제 덱에서 플레이어당 **1장을 무작위로** 배분한다 (2장 중 1장 선택하는 절차 없음). MVP 단순성 우선이며, 추후 "2장 중 선택" 방식으로 바꾸더라도 setup.ts 내부 배분 로직만 교체하면 되고 GameState 계약(`secretAgendaId: AgendaId | null`)은 유지된다.

### A-8. 유권자 그룹 명칭
- 브리프가 명시적으로 언급하는 유권자 그룹은 §11의 "노동자"뿐이다. 나머지 그룹 구성은 브리프 미명시 사항이다.
- 결정: 48장을 6그룹 × 8장으로 균등 배분한다 — 노동자, 중산층, 자영업자, 청년층, 노년층, 농어촌 (`packages/data/src/tags.ts`의 `VoterGroupKey`).
- 각 유권자 카드는 그룹과 별개로 카드 단위 선호/비선호/직접충돌 태그(§10)를 가지므로, 그룹 소속 자체가 투표 계산에 직접 개입하지 않는다 — 그룹은 §15/§17의 "그룹별 영향력 마커 누적·최종 의제 판정" 집계 단위로만 쓰인다.

### A-9. 공약 반응 태그 = 정책 트랙 파생 10종
- §9의 "유권자 반응"과 §10의 "선호/비선호/직접 충돌 태그"가 참조하는 태그 어휘를 브리프가 별도로 정의하지 않는다.
- 결정: 정책 트랙 5종 × 방향 2종에서 기계적으로 파생되는 10개 태그로 통일한다 (`economy-welfare`/`economy-market` 등, `packages/data/src/tags.ts`의 `PolicyReactionTag`). 공약의 `reactionTag`는 자신의 `policyMove.track/direction`에서 그대로 계산되고, 유권자의 `conflictTag`는 `preferredTag`의 정반대 태그(같은 트랙, 반대 방향)로 계산된다. 태그 어휘를 이원화하지 않음으로써 "공약 정책 트랙과 방향이 같음"(§12)과 "선호 태그와 맞음"(§10) 판정이 동일한 비교 로직을 공유할 수 있다.
